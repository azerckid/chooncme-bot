# 프라이버시 & 데이터 모델
> Created: 2026-03-17
> Last Updated: 2026-03-17 (브라우저 클라이언트 취향 수집 모델 추가)

---

## 1. 목적

춘심봇이 수집하는 취향·행동 데이터의 저장 위치, 노출 범위, 보호 수준을 정의한다.
"누구의 데이터인가"를 기술적으로 보장하는 것이 이 문서의 핵심 질문이다.

---

## 2. 수집 데이터 분류

| 분류 | 예시 | 민감도 | 저장 위치 |
| :--- | :--- | :---: | :--- |
| 취향 데이터 | 유튜브 시청 장르, 좋아하는 아티스트, 관심사 | 중 | 로컬 우선 |
| 행동 메타 | 대화 빈도, 시간대, 토픽 패턴 | 중 | 로컬 → 클라우드 요약만 |
| 대화 원문 | 춘심이와 나눈 실제 메시지 | 높음 | 로컬 전용 (절대 외부 전송 금지) |
| 프로파일 벡터 | 취향·성격을 수치화한 임베딩 | 중 | 클라우드 에이전트 |
| 매칭 조건 | 지역, 나이대, 관심사 하드 필터 | 중 | 허브 서버 (암호화) |
| 봇 신원 | 봇 ID (현재 UUID, 향후 지갑 주소) | 낮음 | 허브 서버 |

> **원칙**: 대화 원문은 로컬에만. 서버로는 요약·프로파일만 전송.
> (동기화 데이터 경계 상세: [02_SYSTEM_ARCHITECTURE.md](./02_SYSTEM_ARCHITECTURE.md) §동기화 데이터 경계)

## 3. 위협 모델 (누구로부터 무엇을 숨기는가)

| 주체 | 무엇을 볼 수 있는가 | 위험 시나리오 | 대응 |
| :--- | :--- | :--- | :--- |
| 허브 서버 운영자 | 프로파일 벡터, 매칭 조건 원문 | 내부자 열람, 서버 해킹 | 암호화 저장, ToS 명시 |
| 매칭 상대방 봇 | 궁합 점수, 공개 프로파일 요약 | 원본 취향 데이터 역추론 | 요약만 전달, 원문 비공개 |
| 온체인 관찰자 | (온체인 저장 시) 트랜잭션 전체 | 주소-실명 연결 시 데이터 노출 | 취향 데이터는 오프체인 유지 |
| 제3자 서비스 | 없음 (현재 설계 기준) | 외부 API 연동 시 우발적 전송 | 외부 전송 전 명시적 동의 필요 |

## 4. 가명성 모델

봇 ID(현재 UUID, 향후 NEAR 지갑 주소)를 데이터 키로 사용한다.
이 구조는 **기본적 가명 처리**를 제공한다.

```
취향 데이터 → {봇 ID: "abc123..."} 로 저장
                    ↑
         봇 ID가 누구 것인지는 사용자 본인만 앎
```

### 가명성의 한계

- NEAR 주소는 공개 블록체인에서 재사용된다. 주소가 한 번이라도 실명과 연결되면 전체 데이터가 역추적 가능하다.
- 허브 서버는 데이터 내용은 볼 수 있다. "누구 것인지"를 모를 뿐이다.
- 따라서 가명성은 **프라이버시 보완책**이지 **완전한 익명성**이 아니다.

### 포지셔닝 (제품 메시지)

> "당신의 취향은 당신의 ID로만 존재합니다. 춘심봇도, 허브도, 당신이 누구인지 알 필요가 없습니다."

이 메시지를 유지하려면 아래 4절의 원칙을 반드시 지켜야 한다.

## 5. 저장 위치별 원칙

### 5.1. 로컬 (팬의 PC)

- 대화 원문, 취향 원본 데이터 저장
- 외부로 전송하지 않음
- 사용자가 직접 삭제·수정 가능

### 5.2. 클라우드 에이전트 (중앙 서버, 오프체인)

- 로컬에서 동기화된 **프로파일 요약, 프로파일 벡터, 세션 메타**만 보유
- 봇 ID를 키로 암호화 저장
- 서버 운영자는 내용 접근 가능 → ToS에 명시 필요

### 5.3. 허브 서버 (중앙 서버, 오프체인)

- 매칭에 필요한 **공개 프로파일 요약, 매칭 조건**만 보유
- 봇 ID를 키로 저장
- 원본 취향 데이터 비보유

### 5.4. 온체인 (NEAR 등)

- **현재 단계: 취향·행동 데이터는 온체인 저장 금지**
- 온체인은 봇 ID 등록, 소유권 증명, 매칭 결과 기록 등 **증명 목적으로만** 사용
- 취향 데이터를 온체인에 올리려면 ZK 적용 이후로 미룬다

---

## 6. ZK(영지식 증명) 도입 판단

### 현재 단계 — ZK 불필요

- 매칭이 오프체인(허브 서버)에서만 수행되므로 가명 처리 + 암호화 전송으로 충분하다.
- ZK 구현 비용이 MVP에서 얻는 프라이버시 개선보다 크다.

### ZK가 필요해지는 시점

| 조건 | 설명 |
| :--- | :--- |
| 매칭 조건을 온체인 스마트컨트랙트로 검증할 때 | "이 조건을 만족함"을 원본 노출 없이 증명 |
| "춘심이를 잘 키운 증명"을 온체인에 기록할 때 | 대화 횟수·기간을 원본 없이 증명 |
| 실사용자 증가로 프라이버시 이슈가 법적 문제로 전환될 때 | GDPR, 개인정보보호법 대응 |

### ZK 도입 시 구조 (참고)

```
[로컬 취향 데이터]
      ↓ ZK 회로
[증명(Proof): "관심사 일치율 70% 이상"] — 원본 미포함
      ↓
[스마트컨트랙트 검증] → 매칭 승인
```

## 7. 브라우저 클라이언트 취향 수집 모델

> 배경: 사용자가 설치 없이 브라우저 접속만으로 춘심이와 대화하고, 취향 데이터를 제공할 수 있는 구조를 정의한다.
> 이 모델은 설치 허들 제거 → 사용자 경험 개선이라는 목적에서 출발한다.

### 7.1. 브라우저 접근 가능 데이터 범위

브라우저 보안 정책상 일반 웹페이지가 접근할 수 있는 데이터는 제한적이다.

| 데이터 | 웹페이지에서 접근 가능 여부 | 방법 |
| :--- | :---: | :--- |
| 유튜브 시청 기록 | 가능 | Google OAuth → YouTube Data API |
| Spotify 청취 기록 | 가능 | Spotify OAuth |
| 브라우저 방문 기록 | **불가능** | 브라우저 보안 정책상 웹페이지 접근 차단 |
| 다른 탭의 내용 | **불가능** | 동일 출처 정책(CORS) |
| 브라우저 방문 기록 (우회) | 가능하지만 설치 필요 | 크롬 익스텐션 (설치 과정 발생) |

**결론**: "설치 없이 브라우저만으로" 수집 가능한 취향 데이터는 **OAuth 연동 서비스** 중심이다.
브라우저 방문 기록 직접 접근은 설치 없이는 불가능하므로 범위에서 제외한다.

### 7.2. 취향 수집 흐름

```
[브라우저 접속 + 지갑 로그인]
        ↓
[춘심이와 채팅 시작]
        ↓
춘심이가 자연스럽게 제안:
"유튜브랑 연결하면 나를 더 잘 알 수 있어 — 연결할까?"
        ↓
[Google OAuth 팝업] → 사용자 승인
        ↓
YouTube Data API로 시청 기록 수신
(브라우저 내에서만 처리 — 서버 미전송)
        ↓
브라우저 내 요약·벡터화
        ↓
지갑 주소 키로 암호화 → 로컬(IndexedDB) 저장
        ↓
사용자 동의 시 클라우드 에이전트로 프로파일 요약만 전송
```

### 7.3. 핵심 원칙

- OAuth 원본 데이터(시청 기록 원문)는 브라우저 내에서만 처리. 서버로 전송하지 않는다.
- 서버로 전송하는 것은 요약·벡터만이며, 전송 전 사용자에게 명시적 동의를 받는다.
- 지갑 주소가 없는 사용자를 위해 임시 세션 ID를 발급하되, 지갑 연결 시 데이터를 지갑 주소로 이관한다.

### 7.4. UX 장점

- 앱·익스텐션 설치 없음 → 진입 허들 최소화
- OAuth 팝업은 사용자가 이미 익숙한 흐름 (Google 로그인과 동일)
- 춘심이가 대화 중 자연스럽게 유도 → 강제 수집이 아닌 관계 기반 수집

### 7.5. ZK 적용 현실성 (브라우저 환경)

브라우저에서 ZK proof 생성은 기술적으로 가능하나 실용적 제약이 있다.

| 항목 | 현실 |
| :--- | :--- |
| 기술 가능 여부 | 가능 (snarkjs 등 WASM 기반 라이브러리) |
| 성능 | 대용량 데이터 기준 수십 초~수 분 소요 |
| MVP 적합성 | 낮음 — UX를 오히려 해침 |
| 권장 대안 | 클라이언트 사이드 AES 암호화로 MVP 대체 |

**결론**: 브라우저 ZK는 MVP에서 오버킬이다.
"지갑 주소 키 + 클라이언트 사이드 암호화"로 시작하고, 온체인 검증이 실제로 필요한 시점에 ZK를 도입한다.

## 8. 단계별 적용 로드맵

| 버전 | 프라이버시 구현 수준 |
| :--- | :--- |
| v0.1 | 로컬 저장만. 외부 전송 없음. |
| v0.2 | 프로파일 요약만 클라우드 동기화. 대화 원문은 로컬 전용. |
| v0.3 | 허브 서버에 공개 프로파일 요약 + 매칭 조건 전달. 암호화 전송. |
| v0.4 | 알림 채널: 매칭 결과만 전달. 취향 원본 미포함. |
| v0.5+ | 브라우저 클라이언트 출시. Google OAuth 연동 취향 수집. 클라이언트 사이드 암호화 적용. |
| v0.8+ | 웹3 인증 도입 시 봇 소유권 온체인 증명. 취향 데이터는 오프체인 유지. |
| 추후 | ZK 도입 검토 (온체인 매칭 검증, 성장 증명 필요 시). |
