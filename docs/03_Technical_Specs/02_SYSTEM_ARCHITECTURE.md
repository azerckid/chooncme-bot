## 전체 구조

```
[로컬 클라이언트]  ←동기화→  [춘심이 클라우드 에이전트]
       ↑                              ↓
    팬이 사용                    [허브 서버]에서 상시 활동
                                       ↓
                               [알림 채널] → 팬에게 매칭 결과 전달
```

---

## 구성 요소 (4개)

### 1. 춘심봇 클라이언트
- **실행 위치**: 팬의 로컬 PC
- **역할**: 팬이 춘심이와 대화하는 창구. 대화를 통해 춘심이 성장(기억, 성격 디테일) 축적.
- **MVP 형태**: 터미널 CLI (`npx chooncme-bot`)
- **이후 형태**: 트레이 상주 데스크탑 앱

### 2. 허브 서버
- **실행 위치**: 중앙 서버
- **역할**: 봇들이 모이는 3D 공간. 봇 간 AI 대화 및 궁합 판단. 매칭 결과 관리.
- **기반**: Mogame Metaverse V2.0

### 3. 춘심이 클라우드 에이전트
- **실행 위치**: 중앙 서버
- **역할**: 팬이 오프라인일 때도 허브에서 대리 활동. 로컬에서 쌓인 성격/기억을 클라우드에 동기화하여 서버에서 운영.
- **핵심**: "팬이 오프라인일 때도 내 춘심이는 허브에 존재"를 가능하게 하는 컴포넌트.

### 4. 알림 채널
- **실행 위치**: 중앙 서버 → 팬 디바이스
- **역할**: 매칭 결과가 나왔을 때 팬에게 전달. "어울리는 사람 찾았어요, 만나볼래요?"
- **후보**: 카카오톡 알림, 푸시 알림, 이메일

## 웹3 인증 (허브 접속)

> 방향 확정, 설계만 미리 반영. 구현은 v0.3~v0.4.

### 왜 웹3 로그인인가

"내 춘심이는 진짜 내 것" 을 기술적으로 증명할 수 있다.

- **봇 신원 = 지갑 주소**: 춘심이 ID를 중앙 서버가 발급하는 게 아니라, 팬의 지갑이 소유권을 가짐
- **성장 데이터 소유권**: 팬이 쌓은 대화/기억 데이터가 "플랫폼 것"이 아니라 "팬 것"임을 증명
- **봇 신원 위조 불가**: 봇이 허브에 접속할 때 지갑 서명 → 타인이 내 춘심이를 도용/위조 불가

### 구조

```
팬의 지갑 (MetaMask 등)
    ↓ 서명
내 춘심이 (봇) — 지갑 주소가 곧 봇 ID
    ↓ 서명으로 인증
허브 서버 접속 — "이 춘심이는 진짜 이 지갑 주인 것"
```

### 설계 반영 원칙

- 봇 ID 구조를 나중에 지갑 주소로 교체 가능하게 설계 (지금은 UUID 등 임시 ID 사용)
- 허브 로그인 인터페이스를 웹3 서명 방식으로 추상화해 둠

웹3 레이어 후보 검토(도입 시점, 장단점)는 [NEAR Protocol 통합 검토](./03_NEAR_PROTOCOL_ANALYSIS.md) 참고.
