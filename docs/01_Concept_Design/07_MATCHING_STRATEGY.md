## 1. 매칭 타입 — 연애에서 커뮤니티 전반으로 확장

매칭의 본질은 **"서로 맞는 사람들을 연결"** 하는 것이다. 연애 매칭과 협업 매칭은 동일한 엔진으로 처리할 수 있다.

| 매칭 타입 | 설명 |
|----------|------|
| 연애·이성 매칭 | 주인의 연애 상대 탐색 |
| 팀원 구하기 | 주인이 함께 일할 사람 찾기 |
| 팀 합류하기 | 주인이 들어갈 팀 찾기 |

매칭 타입과 직군은 **독립 변수**로 처리한다. 단일 매칭 엔진이 모든 조합을 커버한다.

```
(연애) + (아티스트)
(팀원 구하기) + (개발자)
(팀 합류하기) + (디자이너)
  → 모두 동일한 엔진, 다른 파라미터
```

---

## 2. 매칭 흐름 — 하드 필터 + 소프트 궁합

```
1단계  하드 필터    주인이 설정한 조건으로 후보 봇 추림
                   (직군, 기술 스택, 지역, 나이대, 관심사 등)
       ↓
2단계  소프트 궁합  조건 통과 봇끼리 AI 대화로 협업 스타일·성향 탐색
       ↓
3단계  궁합 통과    주인에게 알림 → 주인이 연결 여부 최종 결정
```

봇끼리의 대화 내용은 주인에게 요약본으로만 전달된다.

## 3. 증명된 프로필 — 직군별 포트폴리오 플랫폼 연동

자기 신고가 아닌, 이미 공개된 플랫폼 데이터를 봇이 자동으로 파싱해 프로필을 구성한다. GitHub는 개발자용 포트폴리오 플랫폼의 한 사례일 뿐이며, **모든 크리에이터 직군에는 이미 해당 직군의 "GitHub"가 존재한다.**

```
개발자   → GitHub URL     → 코드·언어·활동성 파싱
아티스트  → ArtStation URL → 스타일·장르·활동성 파싱
음악가   → SoundCloud URL → 장르·재생수·활동성 파싱
         ↓
     단일 매칭 엔진이 동일한 구조로 처리
```

### 직군별 플랫폼 & 추출 데이터

| 직군 | 대표 플랫폼 | 추출 데이터 |
|------|-----------|------------|
| 개발자 | GitHub | 주요 언어·프레임워크, 커밋 빈도, 대표 프로젝트, 오픈소스 기여 |
| 그림·일러스트 | Pixiv, ArtStation, Behance | 팔로워, 작품 수, 스타일 태그, 조회수 |
| 음악·작곡 | SoundCloud, Bandcamp | 장르, 재생수, 팔로워, 발매 이력 |
| 영상·모션 | YouTube, Vimeo | 구독자, 조회수, 업로드 빈도, 채널 주제 |
| UI/UX 디자인 | Behance, Dribbble | 프로젝트 수, 어프리시에이션, 스킬 태그 |
| 사진 | 500px, Flickr | 장르, 좋아요, 활동 빈도 |
| 글·기획 | Brunch, Medium, Substack | 구독자, 발행 주기, 주제 태그 |
| 게임 개발 | itch.io | 출시작, 다운로드 수, 장르 |

### 증명된 프로필의 3축

모든 직군에서 동일한 3가지 축으로 프로필이 구성된다.

```
1. 스킬·스타일   무엇을 하는 사람인가        (하드 필터 기준)
2. 활동성        얼마나 꾸준히 하는 사람인가  (신뢰도 지표)
3. 대표 작업물   실제 결과물이 있는가         (소프트 궁합 판단 재료)
```

## 4. 직군 간 협업 매칭 시나리오

같은 직군뿐 아니라 **직군 간 협업 매칭**이 새로운 가치를 만든다.

```
일러스트레이터 (ArtStation)  +  인디 게임 개발자 (GitHub / itch.io)
  → 게임 아트 협업 팀 구성

작곡가 (SoundCloud)          +  영상 편집자 (YouTube / Vimeo)
  → 유튜브 채널 콜라보 매칭

기획자 (Brunch)  +  개발자 (GitHub)  +  디자이너 (Behance)
  → 사이드프로젝트 풀스택 팀 구성
```

## 5. MVP 범위 및 확장 순서

| 단계 | 범위 |
|------|------|
| MVP | 개발자(GitHub) + 팀원 구하기 매칭 |
| v2 | 직군 확장 (아티스트, 음악가 등) |
| v3 | 직군 간 협업 매칭 |
| v4 | 연애·이성 매칭 추가 |

개발자를 첫 타깃으로 삼는 이유:
- GitHub API가 가장 잘 갖춰져 있어 구현 마찰 최소
- "팀원 구하기" 수요가 명확하고 기존 해결책(LinkedIn, 오픈채팅)이 비효율적
- 초기 커뮤니티를 개발자로 좁히면 PMF 검증이 빠름

## 6. 설계 원칙

- **플랫폼은 선택 사항** — 없으면 수동 입력으로 대체, 여러 개 등록하면 통합 프로필 구성
- **공개 API만 사용** — 각 플랫폼의 공개 데이터에 한정, 프라이버시 침해 없음
- **주인이 항상 최종 결정** — 연결 동의, 공개 범위 모두 주인 승인 후 진행
- **단일 엔진, 다중 파라미터** — 매칭 타입과 직군은 독립 변수, 엔진 중복 구현 금지
