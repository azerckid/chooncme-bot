## 1. 개념 요약

춘심봇이 사용자를 "진짜로 알기 위해" 수집하는 패턴 데이터.
대화를 통해 드러나는 **무의식적 성향**이 핵심 — 사용자가 직접 설정한 조건보다 훨씬 정밀한 프로파일을 만든다.

매칭 품질의 핵심 인센티브:
> **춘심이와 오래, 깊이 대화한 주인 = 봇이 더 정확하게 대리 소셜을 한다.**

## 2. 패턴 레이어 분류

### 2.1 대화 행동 패턴

| 패턴 | 데이터 |
| :--- | :--- |
| 접속 시간대 | 주로 밤 11시~새벽 1시에 켠다 |
| 대화 빈도/리듬 | 매일 vs 주말만 / 짧게 자주 vs 길게 가끔 |
| 메시지 길이 | 한 줄 답장형 vs 장문 서술형 |
| 먼저 말 거는 주제 | "오늘 힘들었어" vs "뭐 해?" vs 근황 공유형 |
| 세션 지속 시간 | 5분이면 끄는 사람 vs 1시간 대화하는 사람 |

### 2.2 감정/심리 패턴

| 패턴 | 데이터 |
| :--- | :--- |
| 감정 상태 분포 | 스트레스 토로 비율, 긍정/부정 텍스트 비율 |
| 힘든 날의 신호 | 특정 단어 패턴, 짧아진 답장, 늦은 접속 |
| 선호 위로 방식 | 공감 원함 vs 해결책 원함 vs 그냥 들어줬으면 함 |
| 반응이 좋은 춘심 스타일 | 장난스러울 때 vs 진지할 때 대화가 길어지는가 |

### 2.3 관심사/가치관 패턴

| 패턴 | 데이터 |
| :--- | :--- |
| 자주 언급하는 토픽 | 일, 학교, 게임, 음식, 특정 인물 |
| 명시적 호불호 | 대화에서 직접 말한 좋아하는 것 / 싫어하는 것 |
| 반응 강도 | 어떤 주제에서 대화가 길어지는가 |

### 2.4 관계 패턴 (매칭에 직결)

| 패턴 | 데이터 |
| :--- | :--- |
| 애정 표현 스타일 | 먼저 챙기는 편 vs 챙김 받는 편 |
| 연락 선호 거리감 | 매일 연락형 vs 자유로운 연락형 |
| 갈등 반응 | 춘심이가 삐진 척 했을 때 달래는가 vs 같이 장난치는가 |

## 3. 기술적 수집 가능 여부

### 3.1 v0.1 CLI에서 즉시 가능한 것 ✅

대화 로그를 로컬에 저장하는 것만으로 아래가 수집된다.

| 수집 항목 | 방법 | 비고 |
| :--- | :--- | :--- |
| 접속 시간대 | 세션 시작 타임스탬프 | `new Date()` 저장 |
| 대화 빈도 | 세션 간 간격 계산 | 로그 비교 |
| 메시지 길이 | `message.length` | 단순 카운팅 |
| 세션 지속 시간 | 종료 시각 - 시작 시각 | |
| 자주 쓰는 단어/토픽 | 키워드 빈도 분석 | 간단한 카운터 or Claude 요약 |
| 감정 신호 (간접) | 부정 키워드 출현 빈도 | 규칙 기반 or Claude 분류 |

**저장 형식 제안:** 로컬 JSON 또는 SQLite (`~/.chooncme/history.db`)

```json
{
  "sessions": [
    {
      "started_at": "2026-03-15T23:14:00",
      "ended_at": "2026-03-15T23:52:00",
      "message_count": 24,
      "avg_user_message_length": 18,
      "topics_mentioned": ["일", "피곤함", "게임"]
    }
  ]
}
```

### 3.2 Claude API 활용으로 가능한 것 ✅

세션 종료 시 또는 주기적으로 Claude에게 요약을 요청.

```
시스템: 아래 대화에서 사용자의 관심사, 감정 상태, 반응 패턴을 JSON으로 분석해줘.
```

→ 구조화된 프로파일 자동 생성 가능. 추가 라이브러리 불필요.

### 3.3 v0.2 이후 가능한 것 (데스크탑 앱 전제) 🔜

| 수집 항목 | 방법 | 조건 |
| :--- | :--- | :--- |
| OS 활동 시간대 | 시스템 idle 감지 | 데스크탑 앱 (Electron/Tauri) |
| 앱 사용 패턴 | OS API | 데스크탑 앱 + 사용자 허가 |
| 캘린더 연동 | Google/Apple Calendar API | OAuth 동의 필요 |
| 칸반 행동 데이터 | 별도 태스크 관리 UI | 섹션 3.4 참고 |

### 3.4 브라우저 히스토리 — 사용자 허락 시 수집 가능 🔒

기술적으로 가능하다. 단, **명시적 동의 필수** + **데스크탑 앱 이후** 고려.

**수집 방법 3가지:**

| 방법 | 설명 | 난이도 | 조건 |
| :--- | :--- | :--- | :--- |
| **OS 파일 직접 읽기** | 브라우저가 저장하는 SQLite 파일 접근 | 중 | 데스크탑 앱 + Full Disk Access 권한 |
| **브라우저 확장** | `chrome.history` API 활용, 확장 → 앱 데이터 전달 | 중 | 사용자가 확장 설치 + 권한 허용 |
| **macOS Full Disk Access** | 시스템 환경설정에서 앱에 전체 디스크 접근 허용 | 하 | 사용자가 직접 권한 부여 |

**브라우저별 히스토리 파일 경로 (macOS):**

| 브라우저 | 경로 |
| :--- | :--- |
| Chrome | `~/Library/Application Support/Google/Chrome/Default/History` |
| Firefox | `~/Library/Application Support/Firefox/Profiles/xxx/places.sqlite` |
| Safari | `~/Library/Safari/History.db` |

> 주의: 브라우저 실행 중에는 파일이 잠겨 있어 직접 읽기 불가. 복사 후 읽는 방식 필요.

**춘심봇에서 가치 있는 정보:**

원문 URL이 아닌 **카테고리화된 패턴**만 추출.

```
자주 방문 사이트 분류 → 관심사 벡터
  예) 게임 커뮤니티 70% + 음악 스트리밍 20% → 게임/음악 관심사 확인

검색 키워드 빈도 → 현재 관심사 트렌드
방문 시간대 → OS 활동 패턴과 교차 검증
```

**도입 판단:**

| 항목 | 평가 |
| :--- | :--- |
| 기술적 가능성 | ✅ 가능 |
| 구현 시점 | v0.3+ 옵션 기능 |
| 민감도 | 높음 — 원문 URL은 서버 전송 절대 금지 |
| 추천 방법 | 브라우저 확장이 가장 현실적 |
| 대비 효과 | 대화 + 칸반 데이터로 이미 충분할 수 있음. 선택 기능으로 제공 |

### 3.5 수집 하지 않는 것 ❌

- 브라우저 히스토리 **원문 URL** (카테고리화 후 원문 즉시 파기)
- 다른 앱의 메시지/채팅 내용
- 위치 정보 (명시적 허가 없이)
- 파일 시스템 내용

## 4. 실행 환경별 수집 가능 데이터 비교

사용자가 어떤 환경에서 춘심봇을 사용하느냐에 따라 수집 가능한 데이터가 근본적으로 달라진다.

### 4.1 환경별 수집 가능 항목

| 항목 | CLI (v0.1) | 데스크탑 브라우저 | 모바일 브라우저 | 데스크탑 앱 | 모바일 앱 |
| :--- | :---: | :---: | :---: | :---: | :---: |
| 세션 타임스탬프 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 메시지 길이/빈도 | ✅ | ✅ | ✅ | ✅ | ✅ |
| Claude 요약 프로파일 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 브라우저 히스토리 | ❌ | ✅ (확장 설치 시) | ❌ | ✅ (SQLite) | ❌ |
| OS 활동 시간대 | ❌ | ❌ | ❌ | ✅ | 제한적 |
| 파일 시스템 접근 | ✅ | ❌ | ❌ | ✅ | ❌ |
| 기기 센서 (가속도 등) | ❌ | 제한적 | ✅ (허가 시) | ❌ | ✅ |
| 캘린더 | ❌ | ❌ | ❌ | ✅ (API) | ✅ (API) |
| 헬스 데이터 | ❌ | ❌ | ❌ | ❌ | ✅ (HealthKit 등) |
| 앱 사용 패턴 | ❌ | ❌ | ❌ | ✅ | 제한적 |
| 백그라운드 상시 수집 | ✅ | ❌ | ❌ | ✅ | 제한적 |
| Push 알림 | ❌ | ✅ (Service Worker) | ✅ (제한) | ✅ | ✅ |

### 4.2 핵심 차이점

**브라우저 히스토리**
- 데스크탑 브라우저: Chrome 확장 설치 시 수집 가능
- 모바일 브라우저: 확장 프로그램 미지원 → **수집 불가**

**백그라운드 실행**
- 데스크탑 앱 / CLI: 상시 실행, 사용자 미접속 상태에서도 수집 가능
- 브라우저(모바일/데스크탑): 탭이 닫히면 수집 중단. Service Worker로 일부 가능하나 iOS Safari는 매우 제한적

**모바일 앱에서만 가능한 것**
- 기기 센서 (이동 패턴, 활동량) → "낮에 활동적인 사람 vs 집순이" 판별
- HealthKit / Google Fit → 수면 패턴, 운동 습관 → 라이프스타일 매칭 지표로 활용 가능

### 4.3 환경 조합별 프로파일 풍부도

```
[최소]  모바일 브라우저    → 대화 데이터만 (공통 최소 단위)
        데스크탑 브라우저  → + 브라우저 히스토리 (확장 opt-in)
        데스크탑 앱        → + OS 활동, 파일 시스템, 브라우저 히스토리
        모바일 앱          → + 헬스 데이터, 센서, 캘린더
[최대]  데스크탑 앱
          + 모바일 앱       → 모든 레이어 수집 가능, 가장 정밀한 프로파일
```

> **핵심**: 어떤 환경이든 대화 데이터만으로도 소프트 매칭의 핵심 재료는 충분하다.
> 추가 환경 연동은 프로파일 정밀도를 높이는 선택 기능(opt-in)으로 제공한다.

## 6. 매칭에서의 활용 구조

```
사용자 A 프로파일                   사용자 B 프로파일
- 접속: 주로 자정 이후              - 접속: 주로 자정 이후        → 라이프스타일 일치
- 감정: 부정 토로 비율 높음          - 감정: 공감형 반응 많음       → 보완 관계
- 스타일: 장문 서술형               - 스타일: 짧고 자주             → 보완 관계
- 토픽: 일/스트레스 70%, 게임 20%   - 토픽: 게임 60%, 일상 40%    → 공통 관심사

                   ↓ 봇끼리 실제 대화로 소프트 궁합 검증
                   ↓
          "어울릴 것 같아요!" → 주인들에게 알림
```

하드 필터(사용자 설정) 통과 후, 이 패턴 데이터가 소프트 매칭의 핵심 재료가 된다.

## 8. 프라이버시 원칙

- 모든 원본 대화 로그는 **로컬에만 저장** (서버 전송 금지)
- 허브에 공유되는 것은 **요약된 벡터/프로파일만** (원문 아님)
- 수집 항목은 사용자에게 명시적으로 고지
- OS 접근 권한은 **명시적 동의 후에만** 활성화
- 브라우저 히스토리는 **카테고리 추출 후 원문 즉시 파기** — URL 자체는 저장하지 않음
- 브라우저 히스토리 수집은 **선택 기능(opt-in)** 으로만 제공
