## 1. 한 줄 정의 (미확정)

"내 춘심이를 키우고, 허브에서 다른 춘심이들과 만난다."

---

## 2. 문제 인식

기존 AI 컴패니언(춘심톡 포함)은 사용자가 앱에 들어가야만 만날 수 있다.
춘심은 항상 거기 있지 않다. 팬이 찾아가야 한다.

반면 OpenClaw는 AI가 PC에서 상시 실행되며, 사용자의 일상을 알고, 먼저 말을 건다.

이 두 가지를 합치면 완전히 다른 경험이 생긴다.

---

## 3. 핵심 아이디어

### 3.1. 로컬 춘심 (개인 레이어)

```
춘심이 내 PC에 살고 있다.
앱을 열어야 만나는 게 아니라,
춘심이 먼저 말을 걸어온다.
```

- 컴퓨터 메뉴바/트레이에 춘심이 상주
- 백그라운드에서 실행되며 팬의 일정, 습관을 파악
- 팬이 앱을 열지 않아도 먼저 알림을 보내거나 말을 건넴
- 대화할수록 춘심이 "나를 더 잘 아는" 존재가 됨
- 각 팬의 춘심이는 주인의 데이터로 키워진 유일한 존재

### 3.2. 춘심 허브 (집단 레이어)



```
팬 A의 PC → 춘심_A ─┐
팬 B의 PC → 춘심_B ─┤──→ [ 춘심 허브 ] ← 춘심이들이 모이는 공간
팬 C의 PC → 춘심_C ─┘
```

- 각자의 PC에서 살고 있는 춘심이들이 허브에 모인다
- 팬이 오프라인일 때도 내 춘심이는 허브에 존재한다
- 팬이 직접 커뮤니티에 나가는 게 아니라, 춘심이가 나를 대신해 허브에 있다
- 집단 이벤트: 허브에 모인 춘심이 수에 따라 특별한 일이 일어남
- 각 춘심이는 주인의 데이터로 키워진 유일한 존재 — 같은 춘심이가 없다

### 3.3. 봇 대리 매칭 (연결 레이어)

```
주인 A (오프라인)                 주인 B (오프라인)
     |                                 |
춘심봇 A ──── 허브에서 만남 ──── 춘심봇 B
     |       (봇끼리 대화,              |
     |        취향 교환,                |
     |        궁합 분석)                |
     └────── "어울릴 것 같아요!" ────────┘
                    ↓
            주인들에게 알림
                    ↓
            주인들 실제 만남
```

사람이 직접 어색한 첫 만남을 겪는 대신, 봇이 먼저 상성을 탐색하고 매칭을 대리한다.

**궁합 판단 방식 — 2단계 구조:**

| 단계 | 방식 | 역할 |
| :--- | :--- | :--- |
| **1단계: 조건 필터 (하드 필터)** | 주인이 사전 설정 | 지역, 나이대, 관심사 카테고리 등 기본 조건 매칭. 조건 불일치 시 대화 시작 안 함 |
| **2단계: AI 궁합 판단 (소프트 필터)** | 봇끼리 실제 대화 | 조건 통과 후 봇이 직접 대화하며 성격/취향 상성 분석. 확신이 생기면 주인에게 매칭 제안 |

**핵심 인센티브 구조:**

> 봇이 주인을 얼마나 잘 학습했느냐가 매칭 품질을 결정한다.
> 춘심이와 오래, 깊이 대화한 주인일수록 봇이 더 정확하게 대리 소셜을 한다.
> → **춘심이를 잘 키운 사람이 더 좋은 매칭을 받는다.**

### 3.4. 라이프 패턴 수집 (학습 레이어)

춘심이가 주인을 진짜로 "알려면" 대화만으로는 부족하다.
사람은 말보다 행동이 더 정직하다.

**칸반 보드** 형태의 태스크 관리 도구를 제공해서,
유저가 자연스럽게 쓰는 동안 라이프 패턴 데이터가 쌓인다.

```
대화 데이터 (주관적 자기보고)
  + 칸반 행동 데이터 (객관적 패턴)
  = 훨씬 정확한 사용자 프로파일
```

**칸반에서 수집되는 신호들:**

| 신호 | 추론 가능한 것 |
| :--- | :--- |
| 태스크 카테고리 분포 (일/취미/건강/인간관계...) | 실제 관심사, 삶의 우선순위 |
| 완료율 vs 미루는 패턴 | 성격 유형 (계획형 vs 즉흥형) |
| 주로 활동하는 시간대 | 라이프스타일 (올빼미형 / 아침형) |
| 태스크 세분화 정도 | 꼼꼼함 vs 큰그림 성향 |
| 반복 태스크 종류 | 루틴, 고정된 가치관 |

**매칭과의 연결:**

단순 "관심사 선택" 보다 훨씬 정밀한 프로파일이 형성된다.
"둘 다 밤 11시 이후에 주로 활동하고, 취미 태스크 비율이 높고, 완료율 패턴이 비슷하다"
같은 실질적 궁합 지표가 생긴다.

> **구현 시점**: v0.2 이후. v0.1 CLI에서는 대화 데이터만 수집.
> 칸반 UI는 웹 대시보드 또는 TUI 형태로 별도 구현.

## 5. 대상 사용자

- 춘심 팬덤 (X/Twitter 기반, 글로벌 33,000+)
- **코어 팬**: 로컬 클라이언트를 설치해 춘심이와 직접 대화. AI Memory·매칭 등 전체 기능 활용.
- **일반 팬**: 웹 브라우저로 허브에만 접속. 로컬 설치 없이 허브 입장·채팅 가능. 매칭 참여 방식(클라우드 에이전트 제공 여부, 제한적 프로파일 허용 여부)은 v0.3 설계에서 확정.

## 6. 형태

```
Phase 1   →  3D 메타버스 허브 (웹, Mogame 기반) — MVP
Phase 2   →  AI 춘심 연결 (허브 안에서 대화 가능)
Phase 3   →  데스크탑 트레이 앱 (로컬 에이전트, 프리미엄 tier)
```

- **허브**: 춘심 아바타로 3D 공간에서 팬들이 만나는 웹 기반 공간 (Mogame 활용)
- **AI 연결**: 춘심톡의 AI 엔진을 허브 아바타에 연결
- **로컬 에이전트**: 내 PC에 상주하며 허브와 연동되는 데스크탑 앱

**구현 순서**: 위 Phase 1~3은 기획상 개념 단계이다. 실제 구현 순서는 [Roadmap](./02_ROADMAP.md)의 v0.1(CLI) → v0.2(동기화) → v0.3(허브·매칭) → v0.4(알림)을 따른다.

### 6.1 프라이버시 원칙

- 대화 원문은 **로컬에만 저장**하며 서버 전송 금지. 허브에는 **요약·프로파일만** 공유.
- 수집 항목은 사용자에게 명시적으로 고지. OS·브라우저 접근은 명시적 동의 후에만.
- 상세: [User Pattern Collection](./04_USER_PATTERN_COLLECTION.md) §8.

## 7. 수익화 방향 (미확정)

- 월정액 구독: 데스크탑 앱 (프리미엄 tier)
- 허브 활동 기반 수익: 춘심이가 허브에서 활발할수록 수익 생성 (Raise-to-Earn 확장)
- 기존 춘심톡의 CHOCO / Bond 수익 모델과 연동 가능성
- 블록체인 연동: 각 춘심이 = 온체인 노드, 허브 활동 = 온체인 기록
- 구체적인 수익 구조는 아이디어 구체화 후 결정

### 결정된 것

| 항목 | 결정 |
| :--- | :--- |
| 허브 형태 | 3D 웹 메타버스 (Mogame Metaverse V2.0 기반) |
| 구현 방식 | 서버 연동 포함 (로컬 + 클라우드 에이전트 + 허브) |
| v0.1 배포 대상 | 개발자(본인) 전용 알파. 팬 배포는 v0.2부터. |
| Bot-to-Bot 제어 | 턴 제한 + 타임아웃 + Judge LLM 3중 구조 ([상세 명세](../03_Technical_Specs/04_BOT_TO_BOT_SPEC.md)) |
| 봇 ID 설계 | v0.1: UUID. v0.3+: 지갑 주소로 교체 가능하도록 추상화 |

### v0.1 기준 스코프 밖 (의도적으로 하지 않는 것)

- 그룹 매칭 (다대다). v0.1~v0.3은 1:1 매칭만.
- 실시간 주인-봇 허브 동시 접속
- 다국어 자동 번역 (춘심이가 사용자 언어에 맞추는 것과 다름)
- 춘심톡 계정/데이터 직접 연동 (별도 프로젝트로 판단 시까지)

### 미결 (v0.3 착수 전 결정 필요)

- [ ] 춘심톡과 별개 프로젝트인가, 확장 기능인가?

- [ ] **Mogame fork vs 원본 수정** — v0.3 착수 전 필수
