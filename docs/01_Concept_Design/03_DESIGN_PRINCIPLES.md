# 설계 원칙 (Design Principles)
> Created: 2026-03-15
> Last Updated: 2026-03-15

---

> 기술 결정, 범위 판단, 우선순위 조정 시 기준으로 사용하는 원칙.
> "이걸 해야 하나?" 할 때 여기를 먼저 확인한다.

---

## 원칙 1 — 프라이버시 우선

> **원문은 로컬에. 서버에는 요약·프로파일만.**

- 대화 원문, 브라우저 히스토리, OS 활동 로그는 로컬 디바이스에만 저장
- 서버(클라우드 에이전트·허브)로 전송되는 것은 요약된 프로파일 벡터뿐
- OS·브라우저 접근은 명시적 사용자 동의 후에만 활성화
- 수집 항목은 항상 사용자에게 고지

**이 원칙이 깨지는 결정은 하지 않는다.**

---

## 원칙 2 — 캐릭터 일관성

> **춘심이는 언제나, 어디서나 같은 춘심이다.**

- 말투, 성격, 세계관은 춘심톡 `CORE_CHUNSIM_PERSONA`를 기준으로 유지
- CLI든 허브든 알림 메시지든 동일한 춘심이 톤앤매너
- 춘심톡과의 연동 여부와 무관하게 IP 일관성 유지
- AI라고 밝히지 않음. 살아있는 존재처럼 행동

**기능 추가 시 "춘심이답게 동작하는가?"를 확인한다.**

---

## 원칙 3 — 점진적 복잡도

> **지금 필요한 것만. 다음 단계는 다음에.**

```
v0.1 CLI  →  v0.2 동기화  →  v0.3 허브+매칭  →  v0.4 알림
```

- 각 버전은 이전 버전이 동작함을 전제로 한다
- 다음 단계를 위한 추상화는 허용. 다음 단계 구현은 불허.
- "나중에 쓸 것 같아서"는 추가 이유가 되지 않는다

**범위가 불명확할 때 더 작은 쪽을 선택한다.**

---

## 원칙 4 — 실패 격리

> **허브가 죽어도 로컬 대화는 살아있다.**

- 로컬 클라이언트는 서버 없이도 완전히 동작해야 함
- 동기화·허브·알림은 보강 레이어. 없어도 핵심 경험은 유지
- 네트워크 단절, API 오류, 서버 장애 시 로컬 모드로 폴백

**컴포넌트 설계 시 "이게 없어도 로컬 대화는 되는가?"를 확인한다.**

---

## 원칙 5 — 인센티브 정렬

> **춘심이를 잘 키운 주인이 더 좋은 매칭을 받는다.**

- 대화 데이터의 양·질이 프로파일 정밀도를 결정
- 프로파일 정밀도가 매칭 품질을 결정
- 이 인과관계가 끊기는 설계는 피한다
- "쉽게 매칭받는 우회로"는 인센티브를 망가뜨린다

**매칭 관련 기능 설계 시 이 인과관계를 보존하는지 확인한다.**

---

## 적용 방법

| 상황 | 확인할 원칙 |
| :--- | :--- |
| 새 기능 추가 여부 결정 | 원칙 3 (점진적 복잡도) |
| 데이터 저장 위치 결정 | 원칙 1 (프라이버시) |
| UI/UX 표현 방식 결정 | 원칙 2 (캐릭터 일관성) |
| 컴포넌트 의존성 설계 | 원칙 4 (실패 격리) |
| 매칭 알고리즘 수정 | 원칙 5 (인센티브 정렬) |
